## 1. Hecho (implementado en el repo)

### Web (`web/`)

- **Rutas** (`App.jsx`): `/login` (ruta explícita), `/registro`, `/vinculacion`, `/onboarding`, `/rrhh/alta`, comodín `/*` con `MvpAccessGate` + `MainWithAuth` (login o shell).
- **Menú público** (`components/layout/PublicAuthMenu.jsx`): Iniciar sesión, Registro, Vinculación, Inicio. Resalta sección según ruta; `active="none"` en bypass o carga.
- **Login** (`LoginScreen`, `LoginRoute.jsx`): DNI + PIN, callable `resolverEmailLoginDni` → luego `signInWithEmailAndPassword`.
- **`MvpAccessGate`**: exige `persona_id` en claims; vinculación, onboarding; **ya no** devuelve `null` en carga: muestra menú + “Cargando…”.
- **Bypass dev** (`VITE_BYPASS_AUTH=true`): se muestra `PublicAuthMenu` en `MobileLayout` cuando aplica, encima del header.
- **RRHH — alta agente** (`AltaAgenteRRHH.jsx`): carga `grupos_de_trabajo` y `cfg_rol` vía `listarColeccion`; selector de rol; envía `role_ids` al callable `rrhhAltaAgente`.
- **Onboarding, shell, servicios** (`authService`, `personaService`, `useAuthClaims`, etc.): según módulos ya integrados en esta rama.
- **Catálogo `cfg_rol`**: alineado en seeds y reglas; roles canónicos `CFG_RRHH`, `CFG_USUARIO`, `CFG_MEDICO`, `CFG_VISUALIZADOR` (ver `scripts/seed-v2/`, `configuracionCatalogos.js`).

### Cloud Functions (`functions/index.js` y asociados)

- **`rrhhAltaAgente`**: requiere `data.role_ids` (mín. 1); valida id en `cfg_rol`, `activo !== false`; persiste en `usuarios_cuenta.role_ids`.
- **`resolveRoleIdsRrhhAlta`**, `COL_CFG_ROL`, `normalizeDni` en servidor: acepta DNI enviado como `string` o `number` (JSON).
- **`resolverEmailLoginDni`**: DNI 6–12, rate limit, `personas` + `usuarios_cuenta` (username email, `auth_uid`, no inactiva / no pend. reg. según reglas del código). Errores → 400/… según `HttpsError`.

### Firestore y scripts

- **Reglas** `firebase-v2/firestore.rules`: ajustes para catálogos/lectura según módulo.
- **Raíz `package.json`**: scripts `seed:cfg`, `firebase:emulators:with-functions`, `dev:set-rrhh-claims`, deploy firestore/functions, etc.
- **`scripts/dev-set-portal-role-rrhh.mjs`**: claim `portal_role: "rrhh"`; requiere `usuarios_cuenta` vinculado a ficha.
- **Semilla `cfg`**: `scripts/seed-v2/seed-cfg.mjs`, `seed-ids.v2.json` actualizados con roles.

### Documentación existente

- Bajo `docs/mvp/`, `docs/v2/`: módulos login, configuración, `UNIFICACION_OTRA_PC_Y_TICKET.md`, handoffs anteriores.

## 2. Lo que falta / deuda (priorice según cierre MVP)

- **E2E manual**: login con emulador o nube; tras alta RRHH verificar en Firestore `usuarios_cuenta.role_ids` y sesión/claims.
- **Datos en emulador Firestore**: sin `personas` + `usuarios_cuenta` coherentes, `resolverEmailLoginDni` responde **400** (fallo lógico, no CORS). Hacer `seed` de prueba o flujo pre-alta + vinculación.
- **Java 21+** para emulador Firestore/Functions: en una PC probada, Java 8 → el emulador no arrancaba. Instalar JDK 21+ y ajustar `PATH` / `JAVA_HOME`, o probar sin emulador (nube).
- **Multi-rol en el form RRHH** (varios `cfg_rol` en un solo alta): backend acepta array; UI aún con un select (opcional).
- **Despliegue**: `npm run firebase:deploy:functions` y `npm run firebase:deploy:firestore` cuando el entorno de nube deba alinear con esta rama.
- **ESLint** en el front: fallos en archivos ajenos al menú (p. ej. efectos con `setState` en `AltaAgenteRRHH`, `Configuracion`, `Perfil`) — pendiente de refactors si exigen la regla `react-hooks/set-state-in-effect`.

## 3. Cómo retomar en otra PC (copia remota)

```bash
git fetch origin
git switch mvp-fase1-onboarding
git pull origin mvp-fase1-onboarding
```

En la raíz y en `web/`:

```bash
npm install
cd web && npm install && cd ..
```

- Copiar **`.env.v2.local`** (no versionado) desde el equipo anterior o rearmar a partir de `.env.v2.example` + credenciales Firebase.
- `web/`: asegurar `VITE_*` apunten al proyecto y, si aplica, emulador de functions (puerto **5002** según `firebase-v2/firebase.json` y `functionsV2.js`).

**Arranque local típico**

```bash
# Terminal 1: emuladores (requiere JDK 21+)
npm run firebase:emulators:with-functions

# Terminal 2: Vite
npm run dev:web
```

Si el emulador no corre, probar el front apuntando solo a **Firebase en la nube** (emulador desactivado en `.env.v2.local`).

## 4. Incidentes / mensajes frecuentes (recordatorio)

- **400 en** `POST …/resolverEmailLoginDni` **con emulador**: normalmente faltan documentos o vínculo en `personas` / `usuarios_cuenta` para ese DNI.
- **CORS** en 404: suele ser función no desplegada o URL errónea, no CORS en sí.
- **Cookie** `_ga_*` “expires” sobrescrito: Analytics; no indica error de la callable.
